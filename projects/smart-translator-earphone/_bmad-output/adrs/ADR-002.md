# ADR-002 — Cloud-first vs On-device-first Execution Model

## Context

The translation pipeline (STT → MT → TTS) can run on cloud services, on-device, or as a hybrid. The choice influences:

- Latency (NFR-1: ≤ 1.5 s P95 cloud, ≤ 2.5 s P95 offline)
- Quality (cloud models are larger and stronger today)
- Cost (cloud is variable per-minute; on-device is one-time download + battery)
- Privacy posture (NFR-4: no raw audio retention server-side)
- Offline support (FR-8)

## Decision

**Cloud-first with on-device fallback.** The default execution model is a **streaming WebSocket session** to the cloud plane, terminated at a Cloudflare Worker that fans out to vendor STT/MT/TTS. The on-device pipeline is invoked when:

1. The user has explicitly enabled **Cloud off** in settings.
2. The device has no network connectivity.
3. A specific language pair is configured to run on-device by default (some pairs perform comparably).

The architecture treats cloud and on-device as **interchangeable engines** behind the engine-router interface (see ADR-004 and architecture §3.4).

## Rationale

- **Quality.** As of 2026, cloud STT and MT remain measurably ahead of on-device for most languages, especially at the long-tail (Vietnamese, Thai, Tagalog, Bengali). Defaulting to on-device would degrade user-perceived quality below competitors.
- **Latency.** Counter-intuitively, cloud is often *faster* than on-device for streaming because vendor endpoints have lower TTFT than `whisper.cpp` cold start on a phone. Cloud meets the ≤1.5 s target; on-device meets ≤2.5 s.
- **Battery.** Cloud streaming uses radio more, but the radio is a fixed cost per session vs. CPU/GPU contention from local inference. Net battery use is comparable.
- **Privacy.** A cloud-first system is acceptable *only because* we have engineered the pipeline to **not retain audio**. NFR-4 is enforced architecturally, not just policy-wise (architecture §6.3).
- **Offline as a feature, not the default.** Many users will be online most of the time. The offline pack covers genuine flight/remote-area cases; making it the default would cost quality for everyone to hedge for a minority.

## Consequences

**Positive**

- Best quality available to most users by default.
- Smaller default install size (no model files required at first launch).
- Cost-routing flexibility: we can A/B vendors per language pair without app updates.

**Negative**

- We pay vendor costs continuously (USD ~0.05–0.15 per active minute). Mitigation: free-tier minute caps + Pro pricing model.
- Privacy-conscious users can perceive cloud-by-default as adversarial, even though we don't retain audio. Mitigation: prominent "Cloud off" toggle in settings; one-time disclosure during onboarding (UX §3.5).
- We bear infrastructure complexity (WebSocket gateway, vendor fan-out, rate-limit, observability).

## Alternatives Considered

- **On-device-first.** Promising privacy story but ships a 1+ GB app and fails the latency bar on mid-range devices. Rejected for v1; likely to revisit as on-device models improve.
- **Hybrid streaming (parallel cloud and on-device, pick first).** Doubles cost and battery without clear win. Rejected.
- **Per-language elective (e.g. EU users always on-device).** Operationally complex at v1 scale. The "Cloud off" toggle covers the hard requirement; a future feature may expose per-language elective routing.

## Revisit Triggers

- On-device models close the quality gap on top corridors (e.g. Whisper-Large-v4 on-device, NLLB-200 3.3B on-device).
- A regulatory environment (EU, specific countries) requires on-device by default.
- Vendor cost increases materially erode unit economics.
