# ADR-007 — SOV streaming pipeline: STT-layer VAD vs MT-layer sentence-buffered

## Context

ADR-006 §Layer 2 prescribes a dual-trigger commit pipeline for
SOV languages on Phase B (Whisper-WASM): an audio-layer VAD and
a text-layer sentence-final-particle (SFP) regex, combined by an
OR-rule. During the planning round on 2026-05-07 the team
considered an alternative architecture: keep Whisper-WASM
chunking the audio at fixed intervals as today, and instead
buffer at the **MT layer** until a flush trigger fires.

This ADR records the comparison and explains why STT-layer VAD
(EXT-09) is the production target, with MT-layer buffering
(EXT-09b) preserved as a ship-fast bridge only.

## Two architectures compared

### Option A — STT-layer dual-trigger (EXT-09, recommended)

```
Audio frames ──▶ [VAD silence detector] ──┐
                                          │
Audio frames ──▶ Whisper-WASM ──▶ Text ──▶│ OR-rule ──▶ Commit ──▶ MT ──▶ TTS
                                          │
Text       ──▶ [SFP regex]       ────────┘
```

Audio chunking is **driven by audio characteristics** (silence
gaps in PCM). Whisper runs on those VAD-determined chunks. After
Whisper outputs text, a fast SFP regex provides a second commit
signal. Either signal triggers commit.

### Option B — MT-layer sentence-buffered (EXT-09b, bridge only)

```
Audio frames ──▶ Whisper-WASM (fixed 4 s chunks) ──▶ Text ──▶ [Buffer]
                                                                  │
                                              ┌───────────────────┤
                                              ▼                   │
                                      [Flush trigger]             │
                                              │                   │
                                              ▼                   │
                                          Commit ──▶ MT ──▶ TTS   │
                                                                  │
                                Triggers (any one):               │
                                - Terminal punctuation `[。！？]`  │
                                - Audio silence ≥ 2 s             │
                                - Buffer reaches 200 chars        │
                                                                  ▼
                                                    Buffer holds until trigger
```

Audio chunking stays at fixed 4 s. Text accumulates in a buffer
at the MT layer. Buffer flushes on any trigger.

## Decision

**Option A (STT-layer dual-trigger, EXT-09) is the production
target.** Option B (EXT-09b) is preserved as a ship-fast bridge
that the team may use **only if** EXT-09 is delayed past a
sprint boundary.

## Rationale — head-to-head comparison

| Dimension | Option A — STT-layer (EXT-09) | Option B — MT-layer (EXT-09b) |
|---|---|---|
| Continuous-speech behavior | Audio VAD detects silences naturally created by breathing / pacing — usually ≥ 250 ms even in lectures | Silence trigger requires ≥ 2 s; lecturers often go 10+ s without one. Falls back to 200-char safety bound → 10–15 s commit delay |
| Dependency on Whisper text quality | Audio VAD is independent. SFP regex is one redundant signal | Entire pipeline depends on Whisper text. Hallucinated SFP causes false flushes; missing punctuation causes long delays |
| Provider portability | Audio VAD specific to `audio-capture.js`; SFP regex shared with Phase A and Phase WSP | Provider-agnostic — works on any STT output |
| Effort | M (~3 days) | S-M (~1.5 days) |
| Bundle impact | ~50 LOC pure JS for energy VAD; no model file | None |
| Risk if Whisper-tiny stays | Audio VAD compensates for hallucination | Hallucination directly degrades quality |
| Risk if upstream provider changes | Audio VAD insulates the commit logic | Buffer logic re-derived if STT chunking changes |
| Latency on typical lecture content (estimated) | 5–7 s (per ADR-006) | 10–15 s worst case (200-char safety bound on continuous speech) |
| Latency on conversational content | 6–8 s | 4–6 s (faster — natural pauses + punctuation hit early) |

The decisive factor is **continuous-speech behavior on the
target use case (YouTube lectures, podcasts, vlog narration).**
Option B's fallback to a character-count safety bound produces
unacceptable latency on this content; Option A's audio VAD
maintains acceptable latency regardless of speech rhythm.

The team initially considered Option B as "BESTEST" because of
its provider-agnostic simplicity. Field analysis (continuous
speech use case + Whisper hallucination characteristic) inverted
the verdict.

## When EXT-09b becomes acceptable as a ship-fast bridge

If EXT-09 implementation slips past sprint week 3, the team may
ship EXT-09b first **with a documented expectation** that
SOV-language users will see degraded latency on continuous
content. The popup must show a status badge ("ja/ko/zh
sentence-buffered mode — latency may exceed 10 s on continuous
content") so users can route to Phase A captions or paid
Whisper API in this scenario.

EXT-09 must still ship within v1; EXT-09b is **not** a
substitute, only a bridge.

## Status

**Accepted.** EXT-09 (Option A) is the v1 production target.
EXT-09b (Option B) is preserved in the backlog as a documented
bridge.

## Consequences

### Positive

- The pipeline degrades gracefully on continuous speech, not
  catastrophically.
- Audio-VAD insulation against Whisper hallucination is a
  durable architectural property — future model swaps (whisper-
  small, whisper-medium) inherit it without changes.
- The provider-agnostic SFP regex (text layer) is reused by
  Phase A and Phase WSP, where audio VAD doesn't apply, giving
  uniform commit semantics across all three providers.

### Negative

- Higher implementation complexity than Option B alone. The team
  must add an energy-based VAD module to `audio-capture.js`.
- Whisper-WASM chunking logic must change from fixed 4 s to
  VAD-driven, which is a non-trivial refactor of the existing
  PR #1 code.
- Mocking VAD in unit tests is harder than mocking text
  buffering — additional Vitest test scaffolding required.

### Risks accepted

- VAD threshold tuning may require empirical iteration during
  EXT-09 implementation. The team commits to running EXT-08.5
  (Whisper-base swap with CER measurement) **before** EXT-09 so
  the threshold is tuned against real per-language CER data.
- If EXT-09 effort exceeds 4 days, EXT-09b is the documented
  bridge; this is a planned escape valve, not a failure.
