# ADR-005 — Audio routing strategy on iOS / Android

## Context

The product captures audio from the user's earphone microphone and plays translated audio back through the same earphone (or sometimes through the phone speaker for the conversation partner). On both iOS and Android this is non-trivial because of:

- **HFP vs A2DP** trade-off on Bluetooth — capturing from the mic forces HFP, downgrading playback quality and disabling A2DP on the same device simultaneously.
- **Acoustic Echo Cancellation (AEC)** is mandatory whenever capture and playback share the same device, otherwise the system "hears itself" and translation collapses.
- **Output port routing** (FR-9: user can route TTS to earphone, phone speaker, or both) must be controlled programmatically.
- **Background capture** is heavily restricted (foreground service of type microphone on Android 14+; background mode entitlement on iOS).

The challenge is to define one cross-platform contract that the React Native bridge exposes, while the native modules implement the platform specifics correctly.

## Decision

A unified `AudioSession` interface in `core/audio/` with the following contract:

```ts
type AudioSessionMode =
  | 'capture-only'      // Lecture mode capturing from device mic, playback only on user's earphone via TTS
  | 'duplex-bt'         // Conversation/Group: capture + playback on the same Bluetooth headset
  | 'duplex-wired'      // Same as duplex-bt but for wired/USB-C earphones
  | 'capture-mic-play-speaker'  // Conversation: capture from phone mic (partner speaks), play on phone speaker
  | 'capture-mic-play-earphone' // Conversation: capture from phone mic, play on user earphone
  | 'capture-earphone-play-speaker'  // Conversation: capture from earphone mic, play on phone speaker (for partner)
  | 'capture-earphone-play-both'     // Same as above but mirror to user's earphone too

interface AudioSession {
  start(mode: AudioSessionMode, lang: LangCode): Promise<void>;
  stop(): Promise<void>;
  setOutputRoute(route: 'earphone' | 'speaker' | 'both'): Promise<void>;
  on(event: 'route-changed' | 'bluetooth-disconnected' | 'mic-blocked', cb): void;
}
```

The native modules implement these modes as documented below.

### iOS implementation

- `AVAudioSession.sharedInstance().setCategory(.playAndRecord, options: [...])`:
  - `duplex-bt`: options `[.allowBluetooth, .defaultToSpeaker]` (mode `.voiceChat` for AEC).
  - `duplex-wired`: same category, no `.allowBluetooth`; the system routes to the wired connection automatically.
  - `capture-earphone-play-speaker`: `.playAndRecord` + `.allowBluetooth` + `.defaultToSpeaker`. Use `overrideOutputAudioPort(.speaker)` after activation.
  - `capture-earphone-play-both`: same as above + duplicate the playback graph in `AVAudioEngine` to the speaker route.
- AEC: `AVAudioEngine.inputNode.isVoiceProcessingEnabled = true`. Note this requires `.voiceChat` mode and only works with the system AEC pipeline.
- Background mode: enable the `audio` background mode in Info.plist; require the user-visible mic indicator (system-enforced on iOS 14+).

### Android implementation

- For Bluetooth capture: `AudioManager.startBluetoothSco()` and wait for `ACTION_SCO_AUDIO_STATE_UPDATED` to confirm SCO is active. Capture via `AudioRecord` with `MediaRecorder.AudioSource.VOICE_COMMUNICATION` (engages platform AEC where available). Playback via Oboe in low-latency mode.
- For wired/USB-C: Use Oboe for capture and playback in low-latency mode; AEC via the platform `AcousticEchoCanceler` effect on the input.
- Output routing on `Build.VERSION.SDK_INT >= 31` uses `AudioManager.setCommunicationDevice(AudioDeviceInfo.TYPE_BUILTIN_SPEAKER | TYPE_BLUETOOTH_SCO | …)` to override route programmatically. Pre-31 we fall back to `AudioManager.setSpeakerphoneOn()`.
- Foreground service of type `microphone` (Android 14+) is started before capture and stopped after.

### Engine-router interaction

When the user is in conversation mode, the UI prompts the user to choose whether the partner is also wearing the app's earphones. That choice picks the AudioSessionMode:

- **Both wearing app (group mode)**: `duplex-bt` or `duplex-wired` per device.
- **Only user wearing earphones**: `capture-earphone-play-speaker` (TTS for partner via phone speaker; user's own translation feedback on earphone).
- **User on phone mic, partner on phone speaker**: `capture-mic-play-speaker`.

## Rationale

- This is the place where the highest fraction of platform-specific bugs in any RN audio app come from. Defining the cross-platform contract up front avoids ad-hoc routing rules creeping into UI code.
- Listing all eight modes explicitly forces us to confirm at design time that every UX in the PRD has a corresponding audio routing.
- AEC is mandatory whenever we have the same physical device on both ends; the contract enforces voice-processing where it's needed.

## Consequences

**Positive**

- One TS interface for the entire RN tree. No `Platform.OS === 'ios'` checks scattered through feature code.
- Easier to mock (`MockAudioSession`) for unit tests of the engine router and session manager (Phase 4 implementation does this).
- Easier to swap native implementations (e.g. migrate from `expo-av` to a custom Oboe-based pipeline) as the product matures.

**Negative**

- Eight modes to maintain. Some collapse into the same code path (e.g. `duplex-bt` and `duplex-wired` differ only in audio session options).
- AEC quality varies by device; we ship a "no AEC" fallback for devices where the platform AEC fails (older Android tablets in particular).

## Alternatives Considered

- **Use a third-party real-time-audio library** (e.g. `react-native-webrtc`). Heavyweight; built for video calls, not for low-latency mic-to-cloud streaming. Rejected.
- **One mode (`duplex` only)** and let the OS figure out routing. Rejected — the PRD requires explicit routing options (FR-9).
- **Use Expo `expo-av` as the only audio API.** `expo-av` is sufficient for capture/playback but not for AEC and not for fine-grained routing. We use it as a thin compatibility shim during early scaffolding (Phase 4 stage 1) and replace with native modules as we go.

## Revisit Triggers

- Apple introduces a higher-quality Bluetooth audio profile (e.g. LE Audio with bidirectional 24-kHz capture) — we re-evaluate the HFP downgrade workaround.
- Android adds first-class APIs for simultaneous A2DP playback + HFP capture — we eliminate the music-quality downgrade complaint.
- Platform-specific bugs become unmanageable; we adopt a different audio framework.
