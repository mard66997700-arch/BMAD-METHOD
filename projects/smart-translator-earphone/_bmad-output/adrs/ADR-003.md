# ADR-003 — Streaming STT vs Batch STT for lecture mode

## Context

Lecture mode (FR-2) requires a continuous transcript with translation lag stabilizing at ≤ 3 s. The mode can be implemented two ways:

1. **Streaming STT** — keep a single STT session open; receive interim and final transcripts as audio streams in.
2. **Batch STT** — chunk audio into ~5–10 s segments and send each as a discrete recognition request.

Lecture mode is the **only** PRD-listed mode where this is a real choice; conversation mode (FR-1) requires streaming for its <1.5 s latency target with no acceptable batch alternative.

## Decision

**Streaming STT for both conversation and lecture modes.** The same underlying STT-session abstraction is used; only the orchestration differs (see architecture §3.3 chunker and §3.4 engine router).

## Rationale

- **Latency budget.** Batch with 5 s windows yields a worst-case 5–8 s lag end-to-end; the PRD target is ≤ 3 s lag stabilization.
- **Quality.** Streaming engines maintain a longer attention context across utterance boundaries, yielding better punctuation and proper-noun handling than independent batch chunks.
- **Cost.** Streaming STT is 10–25% more expensive per-minute than batch on most vendors (Deepgram is the exception; streaming and batch are within 5%). The latency win outweighs the cost differential at our subscription pricing.
- **Operational simplicity.** Reusing the conversation-mode pipeline means one set of integration tests, one set of vendor fallbacks, one streaming protocol.

## Consequences

**Positive**

- Hits PRD lecture-mode latency target.
- Single code path for STT across modes; less bug surface.
- Better transcript quality due to cross-utterance context.

**Negative**

- Slightly higher per-minute cost than the batch alternative; absorbed in unit economics.
- The streaming session must be resilient to long-running connections (1+ hour lectures); we must implement explicit session-rotation at 30-minute boundaries to avoid vendor-side timeouts.

## Alternatives Considered

- **Batch with overlapping windows.** Reduces lag below pure-batch but adds dedup logic and still falls short of streaming.
- **Hybrid: streaming for the leading edge + batch for transcript clean-up.** Elegant but doubles the integration surface; rejected for v1.

## Revisit Triggers

- Vendor pricing changes that make streaming materially more expensive than batch.
- Vendor reliability issues with 30-min+ streaming sessions across lecture-length content.
