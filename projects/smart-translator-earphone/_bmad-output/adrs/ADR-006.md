# ADR-006 — SOV-language handling: chunking, commit-delay, and caption-only mode

## Context

Real-time translation of **subject-object-verb** languages (Japanese,
Korean, Chinese-as-spoken, German subordinate clauses) is
fundamentally harder than SVO languages (English, Spanish, French)
in any streaming pipeline. Three concrete failure modes:

1. **Verb-final ambiguity.** A Japanese sentence
   `私は寿司を…` ("I … sushi") can end with `食べたい` (want to
   eat), `食べない` (won't eat), `食べた` (ate), or `食べすぎた` (ate
   too much). Committing the translation before hearing the verb
   is **always wrong half the time**.
2. **Negation bomb (Korean specific).** `아주 좋아…` sounds
   like "I really like…" — until `…하지 않아요` is appended at
   the end, flipping the meaning to "I don't really like." TTS
   that has already played the affirmative cannot be retracted.
3. **German subordinate clauses.** `Ich weiß, dass er Sushi
   essen will` — the verb `will` lands at the very end of the
   subordinate clause. Same problem as Japanese.

This is a well-known **open problem** in NLP. The state of the art
(CUNI SimulStreaming at IWSLT 2025, arxiv 2503.13745) operates on
JA / ZH only at "high-latency 4–5 s" mode; "low-latency 500 ms"
mode is not viable for these languages.

The product targets top-10 source languages including ja, ko, zh.
We must define how the codebase handles them without pretending
the problem is solved.

## Decision

A **three-layered strategy**, each layer applies based on the
active provider:

### Layer 1 — Phase A (YouTube captions): mostly pre-solved, with one edge case

YouTube's caption track contains **fully transcribed sentences
with timestamps** for VOD videos with manual or pre-processed
captions. When the user is on a YouTube watch tab and our
`youtube-captions` provider is active, we typically never see a
half-sentence — `parseJson3Events()` returns complete cues, and
`findEventAt(t)` only emits a cue when its `startMs` window is
entered.

**Edge case — live streams with auto-captions.** YouTube's
live-stream auto-caption pipeline runs an internal streaming
ASR which **does** cut mid-sentence on JA / KO / ZH (same root
cause as Phase B). Detection is straightforward: in
`ytInitialPlayerResponse`, the relevant track has
`captionTrack.kind === 'asr'` (auto-generated) and
`videoDetails.isLive === true`. When **both** flags are set, the
`auto` provider should bypass Phase A and route to Phase B.

This is implemented as story **EXT-08.7** (effort S). The story
must include a defensive AC: when `ytInitialPlayerResponse`
fields are missing or undefined (which happens because this is
an undocumented internal API), the fallback must default safely
to Phase B without throwing.

**Implication:** for VOD content (≈99 % of typical viewing),
Phase A handles SOV better than any STT provider. Live JA/KO
content (≈1 %) routes to Phase B and is governed by Layer 2.
Recommend Phase A as the primary path on YouTube for ja / ko /
zh VOD content; live content runs the Layer 2 pipeline.

### Layer 2 — Phase B (Whisper-WASM): dual-trigger commit (audio VAD + text SFP)

For non-YouTube tabs (Phase B), the current pipeline cuts every
4 s regardless of speech boundaries. This is acceptable for SVO
languages but breaks SOV. We adopt a **dual-trigger commit
pipeline** governed by two **independent** signals combined with
an OR-rule:

- **Audio-layer signal — VAD silence detection.** A lightweight
  energy-based VAD running over the captured PCM closes a chunk
  when it observes a continuous silence ≥ 250 ms *and* the chunk
  is at least 1 s long. This signal is **language-agnostic** and
  **does not depend on STT output quality.** This is critical
  for **continuous speech** (lectures, podcasts, vlog
  narration) where the speaker may not pause for ≥ 2 s for many
  seconds at a time.
- **Text-layer signal — sentence-final particle (SFP) regex.**
  After Whisper-WASM emits a chunk's text, a tiny regex pass
  runs against the trailing 100–200 chars to detect SFP
  patterns: `/(です|ます|だ|だった|だろう)$/` for JA,
  `/(요|입니다|니다|습니다)\??$/` for KO, terminal punctuation
  `[。！？]` for ZH, and verb position heuristics for DE. ~1 ms
  per chunk; no new model.
- **OR-rule for commit.** Commit when **either** signal fires:
  audio silence OR SFP regex hit. This is robust against the two
  failure modes:
  - Whisper-tiny (and even Whisper-base) **hallucinate** SFP
    tokens like `です。` when uncertain, especially on noisy or
    rare-vocabulary content (CER 32.7 baseline on JA per arxiv
    2412.10705). Audio silence detection catches these.
  - Continuous speakers don't produce ≥ 250 ms silences for
    many seconds. SFP regex catches sentence ends even when
    silence trigger is absent.
- **No partial-translation TTS.** Do not feed partial transcripts
  into MT for SOV languages. Wait for committed chunks only.
  This raises latency by 200–800 ms but eliminates negation-bomb
  failures.

**Why this depends on EXT-08.5 (Whisper-base swap) running
first.** With Whisper-tiny CER 32.7 on JA, hallucination
frequency makes the SFP signal noisy enough that early field
reports may show false flushes. EXT-08.5 substitutes Whisper-base
(CER 20.2 baseline) for ja/ko/zh, which reduces hallucination
to a level where the OR-rule with audio silence is robust. EXT-09
(this layer's implementation) **must not start before EXT-08.5
ships and per-language CER is measured.** This is recorded as a
hard ordering dependency in `epics-and-stories-v2.md`.

Cost: latency for ja/ko/zh on Phase B grows from ~5 s steady-state
to ~6–7 s. This is documented in `prd-v1-free-only.md` NFR-LATENCY
as a per-language exception.

**Alternative considered — MT-layer sentence-buffered mode.** A
simpler approach (text-only buffering at the MT layer, flushing
on terminal punctuation OR audio silence OR 200-char safety
bound) was evaluated. It was **rejected as the production
target** because the safety-bound trigger devolves to 10–15 s
of latency on continuous speech (lectures, podcasts) — exactly
the YouTube use case the product targets. It is preserved in the
backlog as **EXT-09b** (optional, S-M effort) for use as a
**ship-fast bridge** if EXT-09 is delayed; not a permanent
substitute. See `adrs/ADR-007.md` for the full architectural
comparison.

### Layer 3 — Caption-only mode (v1.1)

For ja/ko/zh users who prefer correctness over latency, **disable
TTS entirely** and rely on the visible transcript panel + dual-ear
stereo for the original audio only. This shifts the user from
"listen to the translation" to "read the translation while
hearing the original."

**Trigger:** popup checkbox "Caption-only mode (recommended for
Japanese/Korean/Chinese)" — checked by default when source
language is ja/ko/zh.

This matches the architectural choice of high-end translation
glasses (Leion Hey2, RayNeo) which **prefer text on display over
TTS** for SOV languages exactly for this reason: text can be
silently corrected as the verb resolves, but speech cannot be
unspoken.

## Status

**Accepted.** Layer 1 is mostly shipping in PR #1; Layer 1 edge
case (live + auto-captions) added as story **EXT-08.7** (S, with
defensive AC for missing fields). Layer 2 implementation is
story **EXT-09** (M, dual-trigger commit) and depends on
**EXT-08.5** (Whisper-base swap, S) which must run first to
measure baseline CER. Optional **EXT-09b** (MT-layer
sentence-buffered, S-M) is preserved as a ship-fast bridge —
not a permanent substitute. Layer 3 is deferred to v1.1 (story
TBD; depends on EXT-02 popup polish landing first).

## Consequences

### Positive

- Sets explicit, defensible quality bars per language. Users on
  ja/ko/zh see a small "Optimized for SOV" badge in the popup
  rather than wondering why translations are slow.
- Aligns with industry practice (Leion Hey2 / RayNeo / Azure
  Speech Translation all use commit-delay for SOV).
- Caption-only mode is a **product differentiator** — most
  competitors force TTS even when it's wrong.

### Negative

- Latency target in NFR-LATENCY needs a per-language footnote.
  PRD updated.
- Bundle size grows ~75 MB (Whisper-base for ja/ko/zh per
  EXT-08.5). We bundle Whisper-base lazily and only download
  when source language is detected as ja/ko/zh, mitigating the
  hit for SVO users.
- Energy-based VAD is added to `extension/lib/audio-capture.js`.
  This is implemented in pure JS (no model file) using simple
  RMS thresholding; ~50 LOC, no bundle impact.
- Caption-only mode breaks the "dual-ear" tagline for ja/ko/zh
  users. We compensate with marketing copy explaining the trade.

### Risks accepted

- We are **not solving** the SOV problem; we are managing it. A
  user who watches a Japanese video with a long verb-final
  sentence still loses some quality. This is acceptable for v1
  given the entire industry has the same constraint.
- Anticipation models (predict verb from context) are out of
  scope for v1 — they require a custom-trained MT model and
  inference infrastructure we don't have.
- **Whisper hallucination on rare-vocabulary content** —
  identified during this ADR's review pass. Whisper-tiny tends
  to emit fabricated SFP tokens (`です。`, `요.`) when uncertain,
  which would cause false text-trigger flushes. Mitigated by
  (a) the OR-rule using audio-VAD as an independent
  cross-check, and (b) the EXT-08.5 → EXT-09 hard ordering
  dependency that requires real CER measurement before
  finalizing the regex thresholds. Field reports during EXT-01
  live verification should explicitly look for this failure
  mode and report it.
- **`ytInitialPlayerResponse` shape drift** — undocumented
  internal API. EXT-08.7 must include defensive optional-
  chaining checks; in addition, a contract test should run the
  parser against a fixed snapshot of the response shape on
  CI to detect drift early.

## Related industry data points (informational)

- **Leion Hey2 / LLVision** (translation glasses, 2025): claims
  < 500 ms latency on EN→ZH and ZH→EN; not published for JA/KO,
  but field reports indicate +200–400 ms on those pairs.
- **CUNI SimulStreaming** (IWSLT 2025): high-latency mode
  (4–5 s) is the only viable JA/ZH config in their academic
  benchmark. Low-latency mode disabled by config for these
  language pairs.
- **Azure Speech Translation:** uses "automatic break points
  optimized per language pair" — undocumented but observable
  in their latency telemetry.
- **Deepgram Nova-3** (2025): 27 % WER reduction for Korean over
  Nova-2 by handling syllable-block segmentation differently.
  No translation; pairs with downstream MT model.

These data points support the three-layer decision above:
industry consensus is "commit-delay for SOV; prefer text over
speech for SOV."
