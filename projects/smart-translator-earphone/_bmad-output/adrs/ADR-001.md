# ADR-001 — Mobile framework: React Native (Expo Bare) vs Flutter

## Context

The product is mobile-first for iOS 15+ and Android 10+ (PRD NFR-2). It needs:

- Native audio capture (Bluetooth HFP, wired, USB-C) and playback through the same headset.
- Native acoustic echo cancellation (`AVAudioEngine` voiceProcessing on iOS, AEC via `WebRTCAudioProcessing` or platform-specific on Android).
- Native bridges to `whisper.cpp` for offline STT.
- A polished, animated UI with split-screen transcripts and live mic-level meters.
- A productive cross-platform stack so two engineers can ship the v1 MVP in ~3 months.

## Decision

**React Native with the Expo Bare workflow and TypeScript.**

## Rationale

| Dimension | React Native (Bare) | Flutter | Native (Swift+Kotlin) |
|-----------|---------------------|---------|------------------------|
| Cross-platform code share | High (~80%) | High (~90%) | 0% |
| Audio-native ecosystem | Mature: `react-native-audio-record`, `expo-av`, `react-native-track-player`, custom Swift/Kotlin modules straightforward | Moderate: audio plugins exist but bridging custom DSP is more work | Best |
| `whisper.cpp` integration | Direct via custom native module (well-documented patterns) | Direct via FFI / platform channel; less ergonomic | Direct |
| Team velocity | TS/JS familiar; faster iteration via Fast Refresh | Dart learning curve | Slowest by far |
| UI fidelity / animation | Excellent (Reanimated v3) | Excellent | Excellent |
| Long-term maintenance | Reactive ecosystem; libraries occasionally break with RN upgrades | More cohesive ecosystem; Google-funded | Highest TCO (two codebases) |
| Mature audio + Bluetooth APIs | Yes (community modules + custom native modules) | Yes (community plugins) | Yes |

The deciding factors were:

- **Native module ergonomics for audio.** The audio pipeline is the riskiest part of the project. RN's bridging to platform code via TypeScript-typed turbo modules is well-documented and exercised by many real-time-audio apps. Flutter's platform channels are workable but every audio-DSP bridging case I could find required more boilerplate.
- **`whisper.cpp` integration patterns.** There are public, working RN integrations of `whisper.cpp` that we can reference (e.g. `whisper.rn`). Flutter integrations exist but are smaller.
- **TypeScript vs Dart hiring pool.** TypeScript is the dominant typed-JS choice; the team's existing skill set leans here.
- **Expo Bare specifically (rather than Expo Managed)** because we **must** ship custom native modules for audio. Managed mode would force us to use only audited modules.

## Consequences

**Positive**

- Faster iteration, especially for UI work.
- One codebase for both platforms.
- Direct path to `whisper.cpp` integration via well-trodden RN patterns.
- TypeScript end-to-end (with the server too — see ADR-002) reduces mental context switching.

**Negative**

- We will own custom Swift and Kotlin code for the audio pipeline; this is ~40% of Epic 1's effort.
- React Native upgrades occasionally require coordinated migration of native dependencies. We accept a periodic upgrade tax (~1 week per minor version, ~2 weeks per major).
- Performance ceiling for the JS thread is below Flutter's; we mitigate by keeping the audio hot path entirely in native.

## Alternatives Considered

- **Flutter.** Strong contender; rejected primarily on the audio-bridging ergonomics and team familiarity. Reasonable people could disagree.
- **Native (Swift + Kotlin).** Best per-platform fidelity; rejected on schedule. The MVP would slip ~2 months. We will keep the option open as a fallback if RN poses platform-specific blockers.
- **Capacitor / Ionic.** Hybrid web; rejected — the audio pipeline (esp. low-latency native capture and AEC) is not credibly buildable on a WebView-based stack at our latency targets.
- **Kotlin Multiplatform Mobile (KMM) + Compose / SwiftUI.** Promising long-term option; rejected today on tooling maturity for our specific audio needs.

## Revisit Triggers

This ADR should be revisited if any of the following are observed:

- React Native fails to hit our latency targets (NFR-1) due to JS-bridge overhead even after the hot path is in native.
- A whisper.cpp integration in RN proves unstable on Android.
- A new generation of cross-platform tools (e.g. KMM with mature audio plugins) becomes credibly better.
